‚ĚĮ What does this mean in verify.py:     def run(self, expr):                    
          self.IN = expr.upper()                                                

‚óŹ It means the Python verifier uppercases all input before parsing, so you can
  type lowercase and it still works ‚ÄĒ e.g. (quote a) becomes (QUOTE A).

  This matches the C64's behavior: the C64 keyboard produces uppercase by
  default, and SectorLISP uses all-caps atoms (NIL, T, QUOTE, etc.). The
  .upper() just saves you from having to type in caps when running tests in
  Python.

‚ĚĮ Should both lisp.bas and verify.py be able to handle multi-line input?        

‚óŹ Short answer: verify.py already handles it ‚ÄĒ \n is less than ' ', so the
  whitespace-skipping loop in read_expr treats newlines as whitespace. You could
   pass a multi-line string to run() and it would parse fine.

  lisp.bas is a different story. C64 INPUT reads one logical line at a time ‚ÄĒ up
   to ~88 characters (two screen lines), which covers the test suite but would
  choke on anything longer. Adding true multi-line input would require a loop
  that accumulates lines until parentheses balance, then passes the concatenated
   string to GOSUB 2000. Doable, but adds ~10‚Äď15 lines of BASIC.

  My recommendation: skip it for now. The test expressions fit in 88 characters,
   and the C64 context doesn't naturally invite long multi-line sessions. If you
   want to test longer expressions you can always pass them through verify.py.
  Worth revisiting only if you find yourself hitting the input limit in
  practice.

  expression and restarts the prompt cleanly.
